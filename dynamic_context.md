# Dynamic Context: Continuous Context Optimization and Pruning

## Core idea

Current LLM serving systems handle context limits by periodically compacting -- when the context window fills up, summarize and restart. This is crude: you lose information in a single lossy step, and the timing is arbitrary (driven by token count, not information value).

Dynamic context replaces periodic compaction with **continuous optimization and pruning** of the KV cache. The context is a living data structure that is constantly evaluated, compressed, and reorganized during inference. Low-value tokens are identified and removed or summarized in real time, keeping the working context small and high-signal without abrupt compaction events.

## Key insight: multidecode makes re-inference cheap

With vLLM-style paged attention and prefix caching, running inference on a sequence that largely overlaps with a previously cached sequence is extremely cheap. Only the changed/new tokens require actual computation -- the rest hits the KV cache. This means:

- **Evaluating the context** (e.g., scoring token importance) can piggyback on normal inference with minimal overhead
- **Re-prefilling after edits** (replacing a span with a summary) is cheap if only a small portion of the sequence changed
- **Speculative context edits** become feasible -- try a compression, check if quality holds, roll back if not

## The hard question: mid-sequence removal

Can we remove tokens from the middle of a cached sequence without invalidating the entire KV cache after the removal point?

### Option A: Remove and re-prefill

Remove the tokens, splice in a summary, then re-prefill the KV cache for everything after the edit point. This is correct but expensive -- the tail after the edit must be recomputed. For edits near the end of the context this is cheap; for edits near the beginning, it is essentially a full re-prefill.

With prefix caching (vLLM), the prefix before the edit is still cached. So the cost is proportional to the length of the suffix after the edit, not the full sequence.

### Option B: Direct KV cache surgery

Remove the KV entries for the pruned tokens and shift subsequent entries. This is **incorrect** in theory -- every token's KV representation was computed attending to the now-removed tokens, so the cached values are stale. But in practice:

- Attention is sparse. If a removed token received near-zero attention weight from subsequent tokens, removing its KV entry changes almost nothing.
- This is testable: measure the output divergence between cache surgery and full re-prefill. If divergence is small for low-attention tokens, cache surgery is a valid approximation.

### Option C: Hybrid

Score tokens by their downstream attention impact. For tokens with negligible attention from later positions, do direct KV removal (Option B). For tokens with significant attention, replace with a summary and re-prefill the suffix (Option A). Amortize the re-prefill cost by batching multiple edits.

## Continuous optimization loop

Run alongside normal inference:

1. **Score**: For each token (or span) in the context, estimate its information value. Possible signals:
   - Attention weight received from recent tokens (low attention = low value)
   - Redundancy with nearby tokens (repeated or paraphrased information)
   - Recency (older context is less likely to be referenced)
   - Semantic importance (task-relevant entities, instructions, key facts)

2. **Plan**: Identify candidates for removal or summarization. Group adjacent low-value tokens into spans for batch compression.

3. **Compress**: Either remove low-value spans outright or replace them with shorter summaries. Summaries could be generated by the same model (cheap via multidecode on the mostly-cached context) or by a smaller distilled model.

4. **Validate**: After compression, run a quick quality check -- does the model's output distribution shift significantly? If yes, roll back that compression.

5. **Update cache**: Apply the edit using the cheapest valid method (direct KV removal vs. partial re-prefill).

## Building on vLLM

vLLM's architecture provides the primitives needed:

- **Paged KV cache**: Tokens are stored in blocks, making insertion/deletion of blocks feasible without copying the entire cache
- **Prefix caching**: Shared prefixes across sequences are cached, so re-prefill after mid-sequence edits only pays for the changed suffix
- **Continuous batching**: The optimization loop can run as a low-priority background task, consuming spare compute between inference requests

The implementation would likely be a vLLM plugin or fork that adds a context manager layer between the scheduler and the KV cache.

## Experiments to run

1. **Attention sparsity analysis**: On long-context tasks (needle-in-haystack, multi-document QA), measure what fraction of context tokens receive near-zero attention. This is the ceiling for free pruning.

2. **Cache surgery fidelity**: Compare outputs of direct KV removal vs. full re-prefill for tokens below various attention thresholds. Quantify the approximation error.

3. **Compression ratio vs. quality**: On a long conversation or document, progressively compress the context and measure downstream task accuracy. Where is the knee in the curve?

4. **Multidecode overhead**: Measure the actual cost of generating summaries via multidecode on a mostly-cached sequence. Is it cheap enough to run continuously?

5. **End-to-end benchmark**: Compare dynamic context against periodic compaction (e.g., Claude's current approach, LangChain summarize-and-restart) on long-running agent tasks. Metrics: task completion rate, information retention, total compute cost.

## Open questions

- What is the right granularity for pruning -- individual tokens, fixed-size spans, or semantic units (sentences, turns)?
- Can the importance scoring be done without a separate forward pass (e.g., using cached attention weights from the last inference)?
- How does this interact with sliding window attention or other sparse attention patterns?
- Is there a theoretical framework for optimal context compression (rate-distortion theory applied to KV caches)?
- Could this enable effectively infinite context by maintaining a fixed-size optimized window that continuously ingests and compresses new information?
